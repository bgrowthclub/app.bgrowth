## 3. Public Interfaces

The formal contracts this milestone establishes or completes — every one of these is what a future second payment provider, or any other caller, is expected to code against:

- **`CommerceEngine`** — `{ orders, pricing, coupons, tax, refunds, webhooks, paymentManager }`. The single commercial layer the rest of the app is ever allowed to call for anything payment-shaped.
- **`PaymentManager`** — `resolvePaymentProfile`, `resolveProvider`, `createCheckout`, `verifyPayment`, `refund`, `cancel`. Resolves a `PaymentProfileId` (+ optional region) to a concrete `PaymentProvider`; nothing else in the app selects a provider.
- **`PaymentProvider`** — `id`, `createCheckout(request)`, `verifyPayment(transactionId)`, `refund(transactionId)`, `webhook(payload: string, signature: string)`, `cancel(transactionId)`. Every future provider (PayPal, Mercado Pago, ...) implements exactly this shape.
- **`OrderService`** — `createOrder`, `getOrderById`, `listOrdersForMember`, `completeOrder`, `cancelOrder`. `completeOrder` is the one orchestration point after payment confirmation, and is idempotent (safe for at-least-once webhook delivery — sequential duplicates, concurrent duplicates within one process, and mismatched-transaction anomalies are all handled explicitly).
- **`AccessService`** — `hasAccess`, `getAccess`, `listAccessForMember`, `grantAccess`. `grantAccess` is called from exactly one place (`OrderService.completeOrder`) for `purchase`-sourced access.
- **`OrderRepository`** / **`AccessRepository`** — the persistence seams `OrderService`/`AccessService` depend on; never database/HTTP details directly.
- **`Product`** gained `basePrice`/`baseCurrency` (renamed from `price`/`currency` — see §8) and a required `paymentProfileId`. A product never references a payment provider directly.
- **`PaymentProfile`** / **`PaymentProfileId`** — named commercial "modes" (`standard`, `membership`, `free`, `enterprise`, `regional`) a `Product` carries instead of a provider name; `mock/mockPaymentProfiles.ts` is the current registry.
- **`WebhookEvent`** / **`WebhookEventType`** — the provider-agnostic shape any provider's webhook payload normalizes into before anything else in Commerce reacts to it.

## 5. Vercel Functions

| Function | Method | Does |
|---|---|---|
| `/api/checkout` | `POST` | Creates an `Order` (via `server/orderService.ts`), starts a Stripe Checkout Session (via `server/paymentManager.ts`), returns `{ checkoutUrl, sessionId, provider, orderId }` |
| `/api/webhooks/stripe` | `POST` | Reads the raw request body (body parser disabled — signature verification needs exact bytes), verifies + normalizes the event via `StripeProvider.webhook`, calls `OrderService.completeOrder` and nothing else |
| `/api/access` | `GET` | Returns a member's `ProductAccess[]` (`?memberId=`) — the browser's only way to read access, since it must never hold `SUPABASE_SERVICE_ROLE_KEY` |

`api/tsconfig.json` type-checks this folder separately (`tsc -p api/tsconfig.json --noEmit`) since the root `tsconfig.json`'s `include` only covers `src/` and does not check `/api/*.ts`.

## 6. Supabase Schema

`docs/database/supabase-schema.sql` — run once in a Supabase project's SQL editor before setting `SUPABASE_URL`/`SUPABASE_SERVICE_ROLE_KEY`:

- **`orders`** — `id` (PK), `member_id`, `items` (jsonb), `purchases` (jsonb), `subtotal`/`discount_total`/`tax_total`/`total` (jsonb `Money`), `status`, `created_at`, `transaction_id`. Indexed on `member_id`.
- **`product_access`** — `member_id` + `product_id` (composite PK), `has_access`, `source`, `granted_at`, `expires_at`.

Both tables have Row Level Security enabled with **no policies** — the anon/public key has zero access by default; every read/write goes through the service role key from `/api/*.ts` only.

**Scope, deliberately narrow:** only Orders and Access persist to Supabase. Products, Product Packages, Builders, and Workspaces are untouched and remain on their existing data sources.

## 7. Stripe Configuration

What needs to be set up in the Stripe Dashboard (Test Mode) before this is live:

1. **API keys** — copy the Test Mode secret key into `STRIPE_SECRET_KEY`.
2. **Webhook endpoint** — register `https://<deployment-domain>/api/webhooks/stripe`, subscribed **only** to `checkout.session.completed` (the one event type `StripeProvider.webhook` currently maps and processes — any other subscribed event type will be rejected with a clear "unhandled event type" error, by design, not a bug). Copy the resulting signing secret into `STRIPE_WEBHOOK_SECRET`.
3. **Checkout mode** — this flow uses hosted Stripe Checkout (`mode: 'payment'`, one-time), not Stripe.js/Elements — no publishable key or embedded form is involved.
4. **Correlation** — `client_reference_id` on the Checkout Session (and matching metadata on its Payment Intent) carries this app's `Order.id`, which is how the webhook maps back to the right order — no separate lookup table needed.

No production/live-mode keys should ever be used until a separate, explicit go-ahead — this RC is Test Mode only.

## 8. Breaking Changes

Relative to what's actually in `main` today (v0.1.0-platform-foundation):

- **`Product.price` / `Product.currency` renamed to `Product.basePrice` / `Product.baseCurrency`.** Every in-repo reader was updated (`PricingCard`, `PurchaseCard`, `CheckoutPage`, `PreviewDialog`, Studio's Pricing tab, `publishValidation`, `checkout.ts`, `publishedCatalog.ts`, mock data) — but any code outside this diff (a stale branch, an external consumer) reading the old field names will break.
- **`Product.paymentProfileId` is now a required field.** Any code constructing a `Product` literal without it will fail to type-check.
- **`/checkout` and `/checkout/success` now require a signed-in member** (wrapped in `ProtectedRoute`) — previously public routes. A guest is redirected to `/login`. Necessary because creating an `Order` requires a `memberId`.
- **`CheckoutSuccessPage` now reads `productId`/`workspaceId` from the URL query string, not router navigation state.** A real Stripe redirect is a full page navigation, which never carries React Router state — this was a required fix, but it means any external link relying on the old `location.state`-based contract no longer works.
- **This app now has a backend.** `vercel.json`'s previous "static rewrite only" deployment model gains three serverless functions. Anything assuming a purely static SPA deploy (no build-time server component) needs to account for this.
- **New required environment variables** for Stripe/Supabase functionality (§4) — the app still builds and runs without them, but `/api/checkout` and `/api/webhooks/stripe` will fail at request time until they're set.

**Not breaking, but notable:** `CheckoutProvider.ts`/`ProviderAdapter.ts` and `LocalOrderRepository.ts`/`LocalAccessRepository.ts` are superseded and unused (see §2) — left in place, not removed, so nothing currently importing them (nothing does) breaks.
